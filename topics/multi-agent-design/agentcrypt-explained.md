# AgentCrypt: AI エージェント同士の会話を「暗号」で守る仕組み

> **論文情報**
> - **タイトル:** AgentCrypt: Advancing Privacy and (Secure) Computation in AI Agent Collaboration
> - **著者:** Harish Karthikeyan, Yue Guo, Leo de Castro, Antigoni Polychroniadou, Udari Madhushani Sehwag, Leo Ardon, Sumitra Ganesh, Manuela Veloso
> - **年:** 2025
> - **URL:** https://arxiv.org/abs/2512.08104

---

## この論文をひとことで言うと

**「AI エージェント同士がやりとりするとき、個人情報や企業秘密が漏れないように、3段階の"鍵のかけ方"を用意した」** という研究。

---

## なぜこの研究が必要なのか

### AI エージェントは「チームで働く」時代になった

最近の AI は、1体で質問に答えるだけではなく、複数の AI エージェントがチームを組んで仕事をするようになっている。例えば：

- **旅行予約:** フライト担当エージェント、ホテル担当エージェント、予算管理エージェントが連携
- **医療:** 検査データを分析するエージェント、診断を提案するエージェント、薬を推薦するエージェントが協力

### チームで働くと「情報が漏れる」

問題は、エージェント同士がやりとりする中身に **あなたの個人情報** が含まれていること。

```
あなた → 旅行エージェントA に「パスポート番号、クレカ番号、行き先」を伝える
         → エージェントA が エージェントB に「この人の予算は○○円、クレカ番号は...」と伝える
         → エージェントB が エージェントC に...
```

この「伝言ゲーム」の途中で、悪意あるエージェントや、サーバーの管理者がデータを盗み見る可能性がある。

### 今までの対策では不十分

従来の方法は「AI モデルに"秘密を漏らすな"と指示する」程度のもの。しかしこれは AI の振る舞いに依存していて、**AI がうっかりミスしたり、攻撃されたりすると簡単に破られる**。

AgentCrypt のアプローチは違う。AI モデルの振る舞いに頼るのではなく、**数学的に破れない暗号** を使って、仕組みのレベルでデータを守る。

---

## AgentCrypt の「3段階の鍵」

この論文の核心は、セキュリティの強さを 3 段階（Level 0 / 1 / 2）で選べるようにしたこと。

全部の通信を最強レベルにすると遅くなるので、**場面に応じて使い分ける** のがポイント。

### Level 0: 普通の暗号化 — 「郵便封筒」

**たとえ:** 手紙を封筒に入れて送る。配達中は中身が見えないが、届いた先では開封して中身を読む。

- エージェント間の通信路を暗号化（HTTPS と同じ仕組み）
- **サーバー（配達員）は中身を見られる**
- 速度はほぼそのまま
- 用途: 天気予報の取得など、秘密でないデータのやりとり

### Level 1: 属性ベース暗号化 — 「特定の人だけ開けられる金庫」

**たとえ:** 金庫に「医師免許を持っている人だけ開けられるロック」をかける。配達員はもちろん、医師でないエージェントも中身を見れない。

- **属性ベース暗号化（ABE）** という技術を使用
- 「医師」「データアナリスト」など、特定の役割を持つエージェントだけが復号（解読）できる
- サーバーはデータを中継するが、中身は読めない
- 速度: 5〜15 倍遅くなる
- 用途: 患者のカルテ情報を医師エージェントにだけ見せたい場合

### Level 2: 準同型暗号 — 「金庫を開けずに中身を計算する魔法」

**たとえ:** 金庫の中に数字が入っている。金庫を開けずに、外からダイヤルを回すと、中の数字が足し算・掛け算される。結果も金庫に入ったまま。最後に鍵の持ち主だけが開けて答えを見る。

これが **準同型暗号（Homomorphic Encryption）** の本質。暗号化したまま計算ができる。

```
普通の計算:
  3 + 5 = 8

準同型暗号での計算:
  暗号化(3) + 暗号化(5) = 暗号化(8)
  ↑ サーバーはこの操作をするが、3 も 5 も 8 も知らない
```

- **誰も中身を見ずに、計算だけが正しく行われる**
- 速度: 50〜200 倍遅くなる（ここが最大の課題）
- 用途: 暗号化した医療データから統計（平均値など）を計算したい場合

## 具体例で理解する

### 例: 暗号化したまま「おすすめ商品」を提案する

```
[あなたのエージェント]
  あなたの年齢 = 17（暗号化して送信）
  購入履歴 = スニーカー, リュック（暗号化して送信）

        ↓ 暗号化されたデータ

[分析エージェント（サーバー上）]
  暗号化(年齢) で「20歳以上か？」を判定 → 暗号化(No)
  暗号化(履歴) で「おすすめカテゴリ」を計算 → 暗号化(スポーツ用品)
  
        ↓ 暗号化されたまま結果を返す

[あなたのエージェント]
  復号して「スポーツ用品がおすすめ」と表示
```

**ポイント:** 分析エージェントは「17歳」という数字も「スニーカーを買った」という事実も **一切知らない**。暗号化されたデータを操作しただけ。

## 3段階の比較まとめ

| | Level 0 | Level 1 | Level 2 |
|---|---|---|---|
| **たとえ** | 封筒 | 特定の人だけ開けられる金庫 | 開けずに中身を計算できる金庫 |
| **サーバーが中身を見れるか** | 見れる | 見れない | 見れない |
| **暗号化したまま計算できるか** | できない | できない | **できる** |
| **速度** | ほぼそのまま | 5〜15倍遅い | 50〜200倍遅い |
| **使いどころ** | 公開情報 | 特定の相手だけに見せたいデータ | 誰にも見せずに計算したいデータ |

## この研究のすごいところ

1. **AI の「お行儀」に頼らない** — 「秘密を漏らすな」と指示する代わりに、数学的な暗号で守る。AI がハッキングされても、暗号が破られない限りデータは安全。

2. **段階的に選べる** — 全部を最強暗号にすると遅すぎて使えない。「この情報は Level 0 でいい」「これは Level 2 で守る」と使い分けられる設計が実用的。

3. **既存のフレームワークに組み込める** — LangGraph や Google ADK といった実際に使われているマルチエージェント開発ツールに統合して動作確認済み。

---

## 現在の課題

- **Level 2 は遅い** — 準同型暗号は計算コストが非常に高く、複雑な処理には向かない
- **暗号化するとデータが膨大になる** — 元のデータの数百倍のサイズになることがある
- **条件分岐が苦手** — 「もし A なら X、そうでなければ Y」のような処理を暗号化したまま行うのは技術的に難しい
- **鍵の管理が複雑** — エージェントが増えるほど、誰にどの鍵を渡すかの管理が大変になる

---

## まとめ: この論文の本質

> **AI エージェントが協力して働く世界では、「誰にも中身を見せずに仕事を進める」技術が必要。AgentCrypt は、暗号技術を使ってそれを実現する最初の包括的なフレームワークである。**

従来の AI セキュリティが「AI に "漏らすな" と言う」アプローチだったのに対し、この研究は **「そもそも漏れようがない仕組みを作る」** という根本的に異なる発想に立っている。これが最も重要なポイント。
